# Blob Emulator (`blobemu`)

A lightweight Azure Blob Storage emulator with automatic tag indexing, purpose-built for local Docker and Kubernetes development of the photo-api.

## Motivation

The photo-api relies heavily on **Azure Blob Storage tag indexing** — the ability to write key/value tags on blobs and later query them with filter expressions such as:

```
@container='images' AND collection='trips' AND album='hong kong' AND isDeleted='false'
```

Microsoft's official local emulator, **Azurite**, does not support blob tag indexing (the `FindBlobsByTags` API returns `NotImplemented`). This makes it impossible to develop or test the core album/collection/photo retrieval logic without a live Azure Storage account.

Additionally, the production Dockerfile uses a `scratch` base image that contains no shell or CLI. The Azure SDK's `AzureCLICredential` falls back to shelling out to `az`, which fails inside `scratch`. Rather than switching to a heavier base image or embedding service-principal secrets, a self-contained emulator removes the Azure dependency entirely for local development.

**blobemu** solves both problems by providing:

1. A REST API that mirrors the subset of Azure Blob operations the photo-api actually uses.
2. Automatic tag indexing backed by SQLite, supporting the same filter expression syntax.
3. A pure-Go, CGO-free binary that runs in a minimal Alpine container.

## Architecture

```
┌─────────────┐  HTTP  ┌──────────┐
│  photo-api  │───────>│ blobemu  │
│ (LocalBlob  │        │ :10000   │
│   Store)    │        │          │
└─────────────┘        └────┬─────┘
                            │
                   ┌────────┴────────┐
                   │   SQLite (WAL)  │  metadata, tags, indices
                   │  blobstore.db   │
                   └────────┬────────┘
                            │
                   ┌────────┴────────┐
                   │   Filesystem    │  raw blob bytes
                   │  /data/blobs/   │
                   └─────────────────┘
```

### Key design decisions

| Decision | Rationale |
|---|---|
| **SQLite via `modernc.org/sqlite`** | Pure-Go driver — no CGO needed, builds with `CGO_ENABLED=0`, runs in Alpine. WAL mode for good concurrent-read performance. |
| **Filesystem for blob bytes** | Avoids bloating the database; blobs can be gigabytes. Path layout: `/data/blobs/{container}/{name}`. |
| **Single binary, single process** | No external dependencies (no separate DB server). Data persists through a Docker volume. |
| **`net/http` Go 1.22+ routing** | Uses the new `mux.HandleFunc("METHOD /path", ...)` syntax. No third-party router needed. |
| **Custom HTTP headers for tags/metadata** | `X-Blob-Tags` and `X-Blob-Metadata` carry JSON-encoded maps on PUT requests, avoiding multipart complexity. |

## Data Model

### SQLite Schema

```sql
CREATE TABLE IF NOT EXISTS blobs (
    id           INTEGER PRIMARY KEY AUTOINCREMENT,
    container    TEXT NOT NULL,
    name         TEXT NOT NULL,
    content_type TEXT DEFAULT 'application/octet-stream',
    size         INTEGER DEFAULT 0,
    created_at   TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    UNIQUE(container, name)
);

CREATE TABLE IF NOT EXISTS tags (
    blob_id INTEGER NOT NULL,
    key     TEXT NOT NULL,
    value   TEXT NOT NULL,
    FOREIGN KEY (blob_id) REFERENCES blobs(id) ON DELETE CASCADE,
    UNIQUE(blob_id, key)
);

CREATE TABLE IF NOT EXISTS metadata (
    blob_id INTEGER NOT NULL,
    key     TEXT NOT NULL,
    value   TEXT NOT NULL,
    FOREIGN KEY (blob_id) REFERENCES blobs(id) ON DELETE CASCADE,
    UNIQUE(blob_id, key)
);

CREATE INDEX IF NOT EXISTS idx_tags_key_value       ON tags(key, value);
CREATE INDEX IF NOT EXISTS idx_blobs_container      ON blobs(container);
CREATE INDEX IF NOT EXISTS idx_blobs_container_name ON blobs(container, name);
```

- **blobs** — one row per blob, uniquely identified by `(container, name)`.
- **tags** — key/value pairs per blob used for indexed queries. Maps 1-to-1 with Azure Blob Index Tags.
- **metadata** — key/value pairs per blob for custom metadata (e.g. EXIF data). Maps to Azure Blob Metadata.
- Foreign keys with `ON DELETE CASCADE` ensure tag/metadata rows are cleaned up with their parent blob.
- The composite index on `tags(key, value)` accelerates the `EXISTS` subqueries generated by the tag filter parser.

### Filesystem Layout

```
/data/
├── blobstore.db          # SQLite database
└── blobs/
    └── images/           # container name
        ├── trips/
        │   └── hong kong/
        │       └── photo1.jpg
        └── vintage/
            └── nature/
                └── landscape.jpg
```

Blob names may contain slashes — intermediate directories are created automatically via `os.MkdirAll`.

## REST API

All endpoints accept and return JSON unless otherwise noted. The server listens on port **10000** by default.

### Health Check

```
GET /_health
```

Returns `ok` (200). Used by Docker health checks and readiness probes.

### Query Blobs by Tags

```
POST /_query
Content-Type: application/json

{
  "query": "@container='images' AND collection='trips' AND isDeleted='false'"
}
```

**Response** (200):
```json
[
  {
    "name": "trips/hong kong/photo1.jpg",
    "container": "images",
    "tags": { "collection": "trips", "album": "hong kong", "isDeleted": "false" },
    "metadata": { "width": "1920", "height": "1080" }
  }
]
```

The query string uses the same syntax as Azure's `FindBlobsByTags` filter expression:
- `@container='value'` — filter by container name
- `key='value'` — filter by tag key/value pair
- Conditions joined by `AND` (case-insensitive)

### List Blobs in Container

```
GET /{container}
```

**Response** (200): JSON array of `BlobInfo` objects with tags populated.

### Download Blob

```
GET /{container}/{blob...}
```

Returns the raw blob data with the correct `Content-Type` header.

### Get Blob Tags

```
GET /{container}/{blob...}?comp=tags
```

**Response** (200):
```json
{ "collection": "trips", "album": "hong kong", "isDeleted": "false" }
```

### Get Blob Metadata

```
GET /{container}/{blob...}?comp=metadata
```

**Response** (200):
```json
{ "width": "1920", "height": "1080" }
```

### Upload Blob

```
PUT /{container}/{blob...}
Content-Type: image/jpeg
X-Blob-Tags: {"collection":"trips","album":"hong kong","isDeleted":"false"}
X-Blob-Metadata: {"width":"1920","height":"1080"}

<binary body>
```

**Response**: `201 Created`

Tags and metadata are passed as JSON-encoded strings in custom HTTP headers. The upload is an upsert — re-uploading the same `container/name` replaces the data, tags, and metadata atomically within a SQLite transaction.

### Set Blob Tags

```
PUT /{container}/{blob...}?comp=tags
Content-Type: application/json

{ "collection": "trips", "album": "hong kong", "isDeleted": "true" }
```

**Response**: `200 OK`

Replaces all tags on an existing blob. The old tags are deleted and the new set is inserted within a transaction.

### CORS

A permissive CORS middleware is applied to all endpoints for local development convenience:
- `Access-Control-Allow-Origin: *`
- All methods and headers allowed
- `OPTIONS` requests return `204 No Content`

## Tag Query Parser

The parser in `query.go` converts Azure-style filter expressions into parameterised SQL.

### Parsing (`ParseTagQuery`)

1. Split the query string on `AND` (case-insensitive, regex: `(?i)\s+and\s+`).
2. For each token, split on the first `=` sign.
3. Strip double-quotes from the key and single-quotes from the value.
4. Classify the condition: if the key is `@container`, mark it as a container filter; otherwise, it's a tag filter.

Returns a `[]Condition` slice:

```go
type Condition struct {
    IsContainer bool   // true when the key is @container
    Key         string // tag key (or "@container")
    Value       string // expected value
}
```

### SQL Generation (`BuildFilterSQL`)

Each condition becomes a SQL clause:

- **Container condition** → `b.container = ?`
- **Tag condition** → `EXISTS (SELECT 1 FROM tags t WHERE t.blob_id = b.id AND t.key = ? AND t.value = ?)`

Clauses are joined with `AND`. The resulting query selects `(id, container, name)` from the `blobs` table.

**Example:**

Input: `@container='images' AND collection='trips' AND album='hong kong'`

Generated SQL:
```sql
SELECT b.id, b.container, b.name FROM blobs b
WHERE b.container = ?
  AND EXISTS (SELECT 1 FROM tags t WHERE t.blob_id = b.id AND t.key = ? AND t.value = ?)
  AND EXISTS (SELECT 1 FROM tags t WHERE t.blob_id = b.id AND t.key = ? AND t.value = ?)
```

Parameters: `["images", "collection", "trips", "album", "hong kong"]`

## Client Integration (`LocalBlobStore`)

The `LocalBlobStore` in `api/internal/storage/local.go` implements the `BlobStore` interface by making HTTP calls to the blobemu service. It is a drop-in replacement for `AzureBlobStore`.

### BlobStore Interface

```go
type BlobStore interface {
    FilterBlobsByTags(ctx, query, containerName, storageUrl string) ([]models.Blob, error)
    GetBlobTags(ctx, blobName, containerName, storageUrl string) (map[string]string, error)
    SetBlobTags(ctx, blobName, containerName, storageUrl string, tags map[string]string) error
    GetBlobMetadata(ctx, blobName, containerName, storageUrl string) (map[string]string, error)
    GetBlobTagList(ctx, containerName, storageUrl string) (map[string][]string, error)
    SaveBlob(ctx, data []byte, blobName, containerName, storageUrl string, tags, metadata map[string]string, contentType string) error
}
```

### Method → Endpoint Mapping

| BlobStore Method | HTTP Request |
|---|---|
| `FilterBlobsByTags` | `POST /_query` with `{"query": "..."}` body |
| `GetBlobTags` | `GET /{container}/{blob}?comp=tags` |
| `SetBlobTags` | `PUT /{container}/{blob}?comp=tags` with JSON body |
| `GetBlobMetadata` | `GET /{container}/{blob}?comp=metadata` |
| `GetBlobTagList` | `GET /{container}` → builds `collection→album[]` map from tags |
| `SaveBlob` | `PUT /{container}/{blob}` with body + `X-Blob-Tags` / `X-Blob-Metadata` headers |

### URL Encoding

Blob names may contain spaces (e.g. `trips/hong kong/photo1.jpg`). The `blobURL` helper splits on `/`, encodes each segment individually with `url.PathEscape`, then rejoins — preserving the path structure while properly encoding spaces and special characters.

### Compile-Time Check

```go
var _ BlobStore = (*LocalBlobStore)(nil)
```

Ensures `LocalBlobStore` satisfies the interface at compile time.

## Application Wiring

In `api/cmd/photo/main.go`, the store selection is driven by environment variables:

```go
var store storage.BlobStore
if blobEmuURL := utils.GetEnvValue("BLOB_EMULATOR_URL", ""); blobEmuURL != "" {
    slog.Info("using local blob emulator", "url", blobEmuURL)
    store = storage.NewLocalBlobStore(blobEmuURL)
} else {
    // ... create AzureBlobStore with DefaultAzureCredential
}
```

- If `BLOB_EMULATOR_URL` is set → uses `LocalBlobStore` (local development).
- If not → falls back to `AzureBlobStore` with `DefaultAzureCredential` (production/Azure).

All HTTP handlers receive the `BlobStore` interface — they are completely unaware of which implementation is active.

## Docker Compose Topology

```yaml
services:
  photo-api:
    environment:
      BLOB_EMULATOR_URL: http://blobemu:10000  # inter-container URL
      STORAGE_URL: http://localhost:10000        # returned in API responses
    depends_on:
      - blobemu

  blobemu:
    build:
      context: ./blobemu
      dockerfile: Dockerfile
      network: host
    ports:
      - "10000:10000"
    volumes:
      - blobemu-data:/data

volumes:
  blobemu-data:
```

### Two Storage URLs

- **`BLOB_EMULATOR_URL`** (`http://blobemu:10000`) — used internally by `LocalBlobStore` to reach the emulator over the Docker network. Its presence also triggers `LocalBlobStore` selection.
- **`STORAGE_URL`** (`http://localhost:10000`) — returned in API response payloads as the base URL for blob paths (so a browser or SPA can fetch images). Points to the port-forwarded address on the Docker host.

### Dockerfile (blobemu)

Two-stage build:

```dockerfile
FROM golang:1.26-alpine AS builder
WORKDIR /build
COPY . .
RUN go mod tidy && CGO_ENABLED=0 GOOS=linux go build -o blobemu .

FROM alpine:latest
RUN apk --no-cache add ca-certificates
WORKDIR /app
COPY --from=builder /build/blobemu .
EXPOSE 10000
VOLUME ["/data"]
CMD ["./blobemu"]
```

Alpine is used as the runtime base (not `scratch`) because SQLite needs a writable filesystem with proper directory support. The `VOLUME` at `/data` is mounted as a named Docker volume to persist data across container restarts.

## Limitations & Future Work

| Limitation | Notes |
|---|---|
| **No authentication** | Intentional — local dev only. Do not expose to untrusted networks. |
| **No blob leasing or concurrency control** | SQLite transactions provide basic atomicity but no lease semantics. |
| **No pagination** | List and query endpoints return all results. Acceptable for development datasets. |
| **No SAS token generation** | Blob URLs are plain HTTP. The SPA fetches directly via the port-forwarded address. |
| **Single-node only** | SQLite is a single-writer database. For multi-replica Kubernetes deployments, consider switching to PostgreSQL. |
| **Subset of Azure API** | Only the operations used by photo-api are implemented. `DeleteBlob`, `CopyBlob`, `ListContainers`, etc. are not present. |

Potential enhancements:
- Add `DELETE /{container}/{blob...}` for soft/hard delete support.
- Implement blob versioning or snapshots.
- Add a web UI for browsing stored blobs and their tags.
- Support `>=`, `<=`, `>`, `<` operators in tag queries for range filtering.
